## Context

`ImageMenu` の BottomSheet では、単体保存と「レス内の画像をすべて保存（件数付き）」が同じ系統のテキスト表示になっているため、操作対象の違いが視覚的に伝わりにくい。
Issue #437 では「すべて保存」行に download + 複数バッジの合成アイコンを追加する要望が明示されており、既存の保存処理ロジックを変更せず UI の判別性を上げることが目的となる。

## Goals / Non-Goals

**Goals:**
- 「レス内の画像をすべて保存」メニューを一目で識別できるアイコン表現を追加する。
- アイコンは小サイズでも判別しやすい形（download + 重なりバッジ）を採用し、BottomSheet 内で視認性を保つ。
- 既存の一括保存/単体保存の動作契約や表示条件を維持する。

**Non-Goals:**
- 一括保存の実行フロー、権限処理、通知文言の変更。
- 単体保存や他メニュー項目の文言・並び順の再設計。
- 新しいアセットファイル追加やテーマ全体のアイコン刷新。

## Decisions

- **「すべて保存」行のみアイコンを追加する**
  - 変更目的は単体保存との識別性向上であり、対象を最小化して影響範囲を限定する。
  - 代替案として全メニュー項目へ統一アイコンを付与する案もあるが、今回の要件を超えるため採用しない。

- **Compose の合成描画で download + filter_none を重ねる**
  - ベースを `download`、右下バッジを `filter_none` として重ねる実装で、Issue 指定の視覚表現を満たす。
  - 代替案として単一のカスタムベクターアセット作成も可能だが、資産追加コストと保守負担を避けるため既存 Material アイコン合成を選ぶ。

- **コピー系2項目にも同じ合成バッジを適用する**
  - 「画像本体コピー」と「画像リンクコピー」に同一のバッジ付きコピーアイコンを適用し、コピー操作群として視認性を揃える。
  - 片方のみアイコンを付けると差異が誤解を生むため、同一デザインを両項目に展開する。

- **既存 API 互換を維持し、メニュー項目の `icon` 表示のみ変更する**
  - アクションIDやコールバック契約を変えず、UI レイヤーの表示責務だけを更新する。
  - これにより ViewModel や保存処理の既存テストへの影響を抑える。

## Risks / Trade-offs

- [Risk] アイコン重なり比率が不適切だと小さい端末でバッジが潰れて見える。
  - Mitigation: 既存メニューアイコンサイズ内でバッジサイズとオフセットを調整し、実機/プレビューで最小幅を確認する。

- [Trade-off] アイコンを動的合成するため、単一アセットより描画実装がやや複雑になる。
  - Mitigation: 合成ロジックを専用の小さいComposableに閉じ込め、他UIに影響を広げない。
